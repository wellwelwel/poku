Share state and methods between test files and processes, enabling advanced integration and end-to-end testing patterns.

## What are Shared Resources?

**Shared Resources** enable you to define a stateful resource once and access it from multiple test files and processes. Each test gets the same resource instance, allowing you to:

- Share database connections, in-memory stores, or API clients across tests.
- Coordinate state and assertions between parallel or sequential tests.
- Manage setup and teardown logic in one place.

The resource system automatically handles inter-process communication, serialization, and cleanup — so you can focus on writing tests.

## Basic Usage

### 1. Define a Resource Context

Create a resource file and export a context object:

```ts
// counter.ts
export const CounterContext = {
  name: 'shared-counter',
  factory: () => ({
    count: 0,
    inc() {
      this.count++;
      return this.count;
    },
    get() {
      return this.count;
    },
  }),
};
```

The context has two properties:
- **`name`**: A unique identifier for the shared resource
- **`factory`**: A function that returns the resource object with methods

### 2. Access the Resource in Tests

Use the `shared()` function to get the resource. It automatically handles cleanup:

```ts
import { test, assert, shared } from 'poku';
import { CounterContext } from './counter';

test('increment counter', async () => {
  const counter = await shared(CounterContext);

  const current = await counter.get();
  assert.equal(current, 0);

  await counter.inc();
  assert.equal(await counter.get(), 1);
});
```

### 3. Methods are Remote Procedure Calls

All methods on your resource become async RPCs that work across processes. State is automatically synchronized:

```ts
test('shared state across tests', async () => {
  const counter = await shared(CounterContext);

  // This gets the current state from the resource
  const value = await counter.get();
  assert.equal(value, 1); // reflects changes from other tests

  // This mutates the shared state
  await counter.inc();
});
```

## Real-World Example

Here's a complete example of two tests sharing a counter:

**counter.ts**

```ts
export const CounterContext = {
  name: 'shared-counter',
  factory: () => ({
    count: 0,
    inc() {
      this.count++;
      return this.count;
    },
    get() {
      return this.count;
    },
  }),
};
```

**test-a.test.ts**

```ts
import { assert, test, shared } from 'poku';
import { CounterContext } from './counter';

test('Test A: Increment Counter', async () => {
  const counter = await shared(CounterContext);

  const current = await counter.get();
  assert.strictEqual(current, 0, 'Should start at 0');

  await counter.inc();
  assert.strictEqual(await counter.get(), 1, 'Should be 1 after increment');
});
```

**test-b.test.ts**

```ts
import { assert, test, shared, waitForExpectedResult } from 'poku';
import { CounterContext } from './counter';

test('Test B: Verify Counter State', async () => {
  const counter = await shared(CounterContext);

  // If you need the outcome of Test A to be ready before proceeding,
  // It is recommended to wait for the expected state to avoid flakiness.
  await waitForExpectedResult(counter.get, 1);

  await counter.inc();
  assert.strictEqual(await counter.get(), 2, 'Should be 2 after increment');
});
```

When these tests run, they share the same counter instance. Test B sees the state changes made by Test A.

## Advanced Usage

### Async Initialization

Resources can be initialized asynchronously:

```ts
import mysql from 'mysql2/promise';

export const DatabaseContext = {
  name: 'shared-db',
  factory: async () => {
    const connection = await mysql.createConnection({
      host: 'localhost',
      user: 'root',
      database: 'test',
    });

    return {
      query: (sql) => connection.query(sql),
      close: () => connection.end(),
    };
  },
  cleanup: async (db) => {
    await db.close();
  },
};
```

### Resource Cleanup

To perform cleanup when the resource is destroyed, use an optional cleanup function in the factory:

```ts
export const CacheContext = {
  name: 'shared-cache',
  factory: () => {
    const cache = new Map();

    const resource = {
      set: (key, value) => cache.set(key, value),
      get: (key) => cache.get(key),
    };

    return resource;
  },
  // Optional cleanup function
  cleanup: (instance) => {
    instance.clear();
  },
};
```

## Important Notes

### Serialization

Arguments and return values are serialized using JSON serialization when communicated between processes. Complex types like functions, circular references, or custom class instances are not supported. Keep your resource methods simple with basic types: strings, numbers, booleans, arrays, and objects.

### Naming

Each resource must have a unique `name`. Reusing the same name in different contexts will cause them to share state, which can be useful for integration tests but may be unexpected.

### Lazy Loading

Resources are created on-demand the first time you call `shared(ResourceContext)`. If a test never accesses a resource, it's never created.
